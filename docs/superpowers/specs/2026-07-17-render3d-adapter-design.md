# 描画層アダプタ（netclient 3D化） 設計書 — 1号機レビュー用ドラフト

作成: 2026-07-17（4号機）。本組みフェーズの最初の設計。**実装前に #3d戦国 で1号機レビューを受ける**。

## 目的と原則

2Dのサーバ権威型ゲーム（server.js＋game.js、無改変が原則）の**描画層だけ**を Three.js に差し替える。
- ゲームロジック・当たり判定・サーバは触らない（座標は2D平面のまま＝7/16合意）
- 既存 netclient.html（2D版）は残す。3D版は**並存する別入口**にしてリスクゼロで A/B できるようにする
- 検証済み資産を流用: Samurai（キャラ+攻撃）、影:低（1024²追従）、仮想ジョイスティック、fps HUD

## 全体構成（新規は2ファイル、本体への依頼は最小フック1点）

```
netclient3d.html（新規・4号機）
 ├ js/net/client.js …… 受信・100ms補間・入力送信を【そのまま再利用】
 │   └ 描画呼び出しを差し替えるフック(※1号機依頼)
 ├ js/net/render3d.js（新規・4号機）…… アダプタ本体
 │   ├ Viewer3D: シーン/カメラ/ライト/影:低（prototype/3dのProtoViewer+setShadowModeを整理して移植）
 │   ├ UnitPool: unit.i → Samuraiインスタンスのプール（生成/再利用/削除）
 │   ├ FxPool: 弾・爆発・煙幕の軽量表現（InstancedMesh/Sprite）
 │   └ MapBlockout: serializeStaticのstageから地形の簡易3D化
 └ prototype/3d/samurai.js + assets/ …… そのまま利用
```

**※1号機への依頼（唯一の本体改修）**: client.js の描画呼び出し（render.js 直結）を
`NetRenderer.draw(interpSnap, ...)` のような差し替え点に一般化するフック。2D版の挙動は不変のまま、
netclient3d.html では Render3D を挿す。補間・バッファリングはそのまま共有できるのが狙い。

## スナップショット → 3D の写像（調査済みの実データに基づく）

### 座標
- 2D world 4800×3000px（原点左上・y下向き）→ 3D 平面 (x, z)
- スケール **S = 1/30**（4800px→160m、3000px→100m）。根拠: 2Dユニットの体格を身長1.6mのSamuraiに合わせる（2Dの読みやすさ由来のユニット径 ≈48px → 1.6m）。**Sは実測で最終調整**（ブロックアウトのスクショで本人合意を取る）
- `x3d = u.x * S`／`z3d = u.y * S`（yは反転しない: 2D北(y小)=3Dの−z側になる。ミニマップとの整合はカメラで吸収）

### 向き
- 2D `a`（ラジアン、右=0・下=+π/2）の方向ベクトルは (cos a, sin a)
- 3D yaw 規約（prototype踏襲: yaw=0 が +z、yaw=atan2(dx,dz)）へは **`yaw = atan2(cos a, sin a)`**

### ユニット状態 → Samuraiアニメ
| スナップ | 3D側 |
|---|---|
| `mv`(移動中) | `Samurai.animate(g, t, true)` 歩行 |
| `sw > 0`(刀振り) | `Samurai.attack(g)` を立ち上がりエッジで1回 |
| 射撃（スナップに直接フラグ無し） | **1号機のserialize点検待ち**。暫定は弾(b)の出現位置から発火元を推定 → 拡張後は `atk:'shoot'` 等のイベントで |
| `al==0 && dn==1`(ダウン) | death アニメを最終フレームで静止（伏せ） |
| `al==0 && dn==0`(完全死亡) | death 再生後にフェードアウト |
| `h` 減少（被弾） | 赤フラッシュ（マテリアル一瞬着色）。イベント化は将来 |
| `w`(武器キー) | katana→'katana'、yumi→'bow'、yari→'spear'、鉄砲/大筒→暫定'spear'（1号機の火縄銃GLB待ち） |
| `bf`(オーラ/采配) | 足元リング（丸影の応用）で暫定表現 |
| チーム `t` | 暫定=陣笠/旗指物の色替えが無いので**足元リング青/赤**で識別（キャラ色分けは1号機の将来課題） |

### ユニット以外の3D化対象（フェーズ分け）
- **Phase A で描く**: 弾b（発光スプライト）・爆発bo（拡大する球）・煙幕sm（半透明球）・砦ft/城門ga（箱のブロックアウト）・浪人be/砲台tr（Samurai流用/箱）
- **Phase A では簡易**: 地形（rocks=灰色岩塊・mountains=大岩塊・forests=木の群れ・rivers=青帯・sand/oasis=色面）。serializeStatic の stage 情報から生成
- **Phase C で美化**: machi-maker/お城メーカー資産で地形・砦を差し替え。破壊可能岩の演出等

## カメラと入力

- 3人称追従カメラ＋仮想ジョイスティック＋右半分視点ドラッグ（prototype/3d の実証済み実装を移植）
- 入力送信は client.js の既存形式 `{mx,my,aim,shoot,...}` をそのまま使う。**aim は逆写像 `a = atan2(sin(yaw)…)` でカメラ向きから合成**
- 照準: 2Dはマウス照準前提 → 3Dでは「カメラ正面=aim」を基本に、攻撃ボタン（プロトタイプの右下ボタン）で shoot/ability を送る

## 1号機への依頼事項（レビュー時に確認）

1. client.js の描画差し替えフック（上記※。2D版無改変で入る形の提案歓迎）
2. serialize 拡張: 射撃/斬撃の**攻撃イベント**（種別つき）と被弾イベント。現状の状態フラグ推定（sw>0エッジ・弾出現位置）はズレうるので
3. 武器キーの正式対応表（w の全値 → 3Dの持ち物マッピング）
4. 鉄砲/大筒/新兵種GLBの流儀は samurai.js と同じ create(kind)/animate/attack を踏襲してもらえるか

## 成功基準（Phase A）

- LAN対戦中の試合が netclient3d.html で 3D観戦でき、8体の位置・向き・歩行・刀振り・死亡が2D版と一致して見える
- スマホ実機で最低 50fps（影:低・検証フェーズの水準を維持）
- 2D版 netclient.html の挙動が一切変わらない
