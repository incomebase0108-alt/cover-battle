# 3D戦況HUD 設計（被弾フィードバック / ミニマップ / 味方の状態）

- 日付: 2026-07-18
- 担当: 4号機
- 対象: `js/net/render3d.js` + `netclient3d.html`（3D描画層のみ。server.js / game.js は無改修）
- 発端: 星野さんの実機評価「格闘が単調」「ダメージを受けていることに気づきにくい」
  「マップ全体図が縮尺で表示されると戦況が把握できる」「味方の状態・位置くらいは」

## 位置づけ

「格闘に駆け引きを作る」という要望を2つに分けたうちの **A（戦況の可視化）**。
B（防御・読み合い・間合い・兵種の個性＝ゲームルールの変更、1号機のテリトリー）は別途設計する。

A を先にやる理由: **被弾も相手の位置も見えない状態では、そもそも読み合いが成立しない**。
A は3D描画層だけで完結するので、Bの設計を待たずに出せる。

## 1. 被弾フィードバック

### 現状の問題
被弾すると当たったキャラ自体が赤く光る（`hu` フラグ → `hitFx`）が、TPSでは自分の背中しか
見えないため気づきにくい。

### 仕様
- **画面の縁が赤く光る**（vignette）。`hu` が立った瞬間に不透明度 0.55 から 0.5秒で 0 へ
- **方向インジケータ**: 被弾方向を、画面中央を基準にした弧で表示。0.8秒でフェード
- 死亡時は縁を濃く出したまま、死亡演出（沈下）に合わせて消す

### 方向の求め方（重要な制約）
スナップの `u[].hu` は **「被弾したか」の真偽だけで、誰が撃ったかの情報がない**。
3D側で次の順に推定する:

1. 直近0.3秒に自分の半径4m以内にいた**敵チームの弾**（`b[]`、`t` でチーム判定）があれば、
   **その弾の位置の方向**。弾は発射元から自分へ飛んでくるので、これが撃たれた方向に近い
   （※スナップの弾は `x,y,t,f,bl,br,ar` のみで**速度を持たない**ため、進行方向は使えない）
2. なければ、自分から最も近い**生存中の敵**の方向（近接のリーチは2〜3mなのでほぼ一致する）
3. どちらも取れなければ方向インジケータは出さず、縁の点滅だけにする

推定で精度が不足する場合の次善策として、1号機へ `hu` に方向（または攻撃者index）の追加を
依頼する。Phase A で `hu` 自体を足してもらったのと同じ流れ。**まずは推定で出して実機で判断する。**

## 2. ミニマップ

### 仕様（2D版 `js/net/render.js` の `_minimap` を踏襲）
- 位置: 右上。幅はワールド比率（4800×3000）を保つ
- **味方は常に表示。敵は視界に入っているときだけ**
  → 索敵の駆け引きを壊さないため、ここは2D版と同じルールを守る
  → 判定は **2D版の `NetRender.visible(u, snap, map, myTeam)` をそのまま呼ぶ**。
    `netclient3d.html` は既に `js/net/render.js` を読み込んでおり、`NetRender` はグローバル。
    森・煙幕の隠蔽と400msのヒステリシスまで込みなので、**同じ判定を二重に実装しない**
- 城（bases）を四角、自分を白い点で表示
- **3D版の追加: 自分の視野方向を扇で描く**（TPSは自分がどちらを向いているか分かりにくい）
- スマホ幅（600px以下）では画面幅の約28%に縮小

### 2D版との違い
2D版はカメラ範囲を白枠で描くが、3Dにカメラ矩形は無いので**視野の扇**に置き換える。

## 3. 味方の状態

ミニマップ上で表現する（別UIは作らない）:
- HPが減ると点の色が変わる（緑 → 黄 → 赤）
- **倒れている味方（`al=false` かつ `dn=true`）は×印**で表示 → 救出に向かえる
- 完全に死亡した味方は表示しない

2D版のミニマップは倒れている味方を描かない。ここは3D版が先行し、良ければ2D版へ展開する。

## データ（既存スナップで足りる）

| 用途 | フィールド |
|---|---|
| 被弾 | `u[].hu` |
| HP | `u[].h` / `u[].mh` |
| 生死・ダウン | `u[].al` / `u[].dn` |
| 位置・チーム | `u[].x` `u[].y` `u[].t` |
| 弾（方向推定） | `b[].x` `b[].y` `b[].vx` `b[].vy` |
| 城 | `Net.map.bases` |

**サーバ改修は不要。** 方向の精度を上げる場合のみ1号機へ依頼する。

## 実装方針

- ミニマップと縁の演出は **DOM + Canvas 2D**（Three.jsのシーンには足さない）。
  3Dの描画負荷に影響させないため、また文字やUIはDOMの方が扱いやすいため
- ミニマップの更新は毎フレームではなく **10Hz** に間引く（スナップが約18Hzなので十分）
- `render3d.js` が肥大化してきたので、HUD関連は **`js/net/hud3d.js` として分離**する。
  `render3d.js` からは「フレームごとに view を渡す」だけの関係にする

## テスト

- 被弾方向の推定は**純粋関数**として切り出し、単体テストを書く
  （弾から推定 / 最も近い敵から推定 / どちらも無い場合に null を返す）
- ミニマップの座標変換（ワールド座標 → ミニマップ内座標）も純粋関数＋テスト
- 描画そのものは実試合のヘッドレスで目視確認（既存の検証ハーネスの型を使う）

## 成功基準

1. 被弾したとき、画面を見ていれば**どちらから攻撃されたか分かる**
2. ミニマップで**味方の位置と、倒れている味方**が分かる
3. 敵はレーダーのように全部は見えない（**視界内のみ**＝索敵の駆け引きが残る）
4. スマホで画面を圧迫しない
5. fpsが落ちない（実機60fps維持）
